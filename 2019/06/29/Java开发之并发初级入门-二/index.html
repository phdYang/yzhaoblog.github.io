线程安全

我们介绍了一些线程的概念，以及我们为什么需要线程，紧接着我们介绍了在Java程序世界中如何创建线程和使用基本的线程操作。

但是，当我们身处多线程环境下会遇到的第一个问题就是就是线程安全。在单线程环境下，所有资源都是该线程持有，但多线程不一样，会出现一些“不可思议”的问题，也就是说，在多线程环境下，线程之间会出现竞态。

在Java语言中，各种操作共享的数据有以下 5类：

  • 不可变
  • 绝对线程安全
  • 相对线程安全
  • 线程兼容
  • 线程对立

比如，对于一个售票系统，当多个客户端同时访问售票系统时，每一个客户端是一个线程，那么对于票的数量来说，就是一个共享资源，此时就会产生线程不安全的问题。

  • 类锁和对象锁不是同一把锁，如（5，7）

Synchronized在JDK中的改进

我们知道，synchronized刚一问世，就是以极其笨重的姿态问世的，JDK几经修改，终于打造了一款高性能的synchronized！

在JDK1.6之前，synchronized一直是以重量级锁来面对世界，在1.6之后，JDK引入了偏向锁和轻量级锁，从而让锁拥有了四种状态：

  • 无锁状态（unlocked）
  • 偏向锁状态（biasble）
  • 轻量级锁状态（lightweight locked）
  • 重量级锁状态（inflated）

他们会随着竞争的激烈程度而逐渐升级。

所以，主要的一个锁优化有以下几个方面：自适应自旋，锁消除，锁粗化，轻量级锁，偏向锁；

我们分别简单讲述一下都是什么作用，接着重点讲解synchronized的锁升级的过程。

自旋

我们知道，对于使用互斥同步方法使得线程进入阻塞状态的开销很大，而共享数据的锁定状态很短，有时候切换的耗时还要大于运行时间。那么，自旋的思想就是让一个线程在请求锁是执行一个忙循环（自旋），如果在这段时间可以获得锁，就可以避免进入阻塞状态。

但是，自旋也是消耗资源的，自旋的默认次数是10。

而在JDK 1.6之后引入了自适应自旋，自旋的时间是由前一次在同一个锁的自旋时间和锁的拥有者决定的。

锁消除与锁粗化

锁消除指的是，判断在一段代码上，堆上的所有数据都不会逃逸出去被其他线程访问到，那么就可以将其视为栈上的数据，是线程私有的。那么，就会忽略所有的同步操作热直接执行

锁消除与逃逸分析都是JDK 1.5以后

逃逸分析（即时编译器优化）：基本方法是分析对象的动态作用域。如果可以证明一个对象不会逃逸到方法或者线程之外，那么可以进行一些优化：栈上分配；同步消除；标量替换

锁粗化指的是，如果虚拟机探测到有这样的一串零碎的操作都对同一个对象加锁，将会把加锁同步的范围扩展到整个操作序列的外部

轻量级锁

轻量级锁始于JDK 1.6，它有别于使用“互斥同步”的重量级锁。其本意是：在没有多线程竞争的环境下，减少重量级锁的性能消耗。

我们在这里简单说一些，什么是轻量级锁，什么情况下使用轻量级锁，如何升级，以及如何加锁和解锁的。更加详细的原理涉及了JVM的知识，请参考《深入理解Java虚拟机：JVM高级特性与最佳实践》。

首先，当线程进入同步块时，如果同步对象没有被锁定，那么虚拟机会在栈帧中创建一个锁记录，然后采用CAS的操作将对象头（Mark Word）更新为指向锁记录（Lock Record）的指针。成功，则获得锁，更新状态为轻量级锁；失败，需要判断是否指针是否指向自身，如果是，则直接进入同步块，否则，就说明这个锁对象已经被线程抢占了，要膨胀为重量级锁。

重量级锁的对象头指针指向互斥量

在解锁的过程中，同样采用CAS替换对象头和它的备份，替换成功，则同步完成；失败，说明有其他线程尝试获取过该锁，那么释放锁的同时需要唤醒被挂起的线程。

轻量级锁认为：对于大部分的锁，在整个同步期间是不存在竞争的。

CAS操作后续详细说明

偏向锁

偏向锁指的是偏向第一个线程。

当锁对象第一次被线程获取的时候，使用CAS把该线程的ID记录在对象头上，持有偏向锁的线程进入同步块，不会进行任何同步操作。当有第二个线程持有偏向锁时，则偏向模式宣告结束。

Synchronized的实现

JVM底层通过监视锁来实现synchronized同步的。首先ACC_SYNCHRONIZED标识，然后使用monitorenter和monitorexit两个字节码指令获取锁和释放锁，这两个字节码都需要一个reference来指明加锁和解锁的对象。使用monitor来作为锁计数器，默认为0。

JVM的优化主要集中在monitor的加锁和解锁上。
